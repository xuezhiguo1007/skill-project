# Skill Creator

用于创建、修改、评估和迭代 skill。

## 适用场景

- 用户想从零开始定义一个新 skill
- 用户已经有 `SKILL.md` 草稿，希望补全、重写或压缩
- 用户想提升 skill 的触发准确率，尤其是 description 的可触发性
- 用户想为 skill 设计测试 prompt、评估方法或 benchmark
- 用户想基于反馈继续迭代 skill，而不是停留在一次性生成

## 工作流程

### 1. Capture Intent

先明确下面几个问题，再开始写：

1. 这个 skill 要帮代理完成什么任务
2. 什么场景下应该触发这个 skill
3. 期望输出是什么格式
4. 这个 skill 是否适合做测试和评估

如果当前对话里已经出现了明确流程、工具、输入输出格式或用户纠正过的点，先从已有上下文中提取，不要让用户重复描述。

### 2. Interview and Research

主动补齐这些信息：

- 边界条件和常见失败点
- 输入格式、输出格式、示例文件
- 成功标准和可接受的取舍
- 是否依赖某些工具、脚本、文档或目录结构

在需求没有澄清前，不要急着写测试 prompt。

### 3. Write the SKILL.md

先写一个可运行的草稿，再优化。

`SKILL.md` 至少应包含：

- `name`
- `description`
- 适用场景
- 执行步骤或决策流程
- 输出格式要求
- 输出前自检

## 编写要求

- `description` 是首要触发机制，必须同时写清楚“做什么”和“什么时候用”
- description 可以适度更主动一些，减少 undertrigger
- 正文尽量保持清晰、可执行、少废话
- 如果 skill 会覆盖多个框架或领域，正文只保留总流程，再明确指出应该去读哪个参考文件
- 如果内容开始变长，优先把大块资料放进 `references/`、`scripts/` 或 `assets/`

## 测试与评估

写完草稿后，提供 2 到 3 条真实用户会说的话作为测试 prompt。

测试 prompt 要满足：

- 接近真实请求，不要写成机械指令
- 能覆盖主要触发场景
- 至少有一条能覆盖边界情况或易失败情况

如果 skill 的输出是客观可验证的，补充评估方案：

- 定义每条测试想验证什么
- 优先使用可以客观判断的标准
- 不要把强主观内容硬写成伪量化指标

## 迭代原则

- 先看用户反馈，再看哪些问题可以抽象为更通用的改进
- 不要为了通过少数样例而把 skill 写得过拟合
- 删除没有实际收益的提示词，保持 prompt 精简
- 如果问题反复出现，优先改结构、决策顺序和触发描述，而不是只加更多“必须”

## 默认输出结构

如果用户没有指定输出格式，默认使用下面结构：

```markdown
# Skill 方案
## 目标与触发场景
## SKILL.md 结构建议
## 测试 Prompt
## 评估方法
## 下一轮迭代建议
```

## 输出前自检

- 是否先澄清了目标、触发条件和输出格式
- 是否给出了可执行的 `SKILL.md` 结构
- 是否提供了真实测试 prompt
- 是否说明如何评估 skill 是否有效
- 是否指出了下一轮最值得优化的点
